# MCP (Model Context Protocol)

**MCP** is a standard way for external processes to expose **tools**, **resources**, and **prompts** to an AI host. In CentrAI Agent, MCP integration lets you connect third-party or internal servers without writing a custom adapter for each API surface—once the MCP client is configured, discovered capabilities map into the same tool execution path as native tools.

## Why use MCP?

- **Separation of concerns** — Tools run in their own process; upgrade or restart them independently.
- **Ecosystem** — Reuse existing MCP servers (filesystem, browser, databases, SaaS connectors).
- **Security boundary** — Network and auth can be constrained at the MCP transport layer.

## Client responsibilities

The runtime’s MCP client should:

1. **Connect** via supported transports (e.g. stdio, HTTP/SSE—as implemented).
2. **Discover** tools and their schemas from the server.
3. **Register** them with the agent under stable names (handle collisions with prefixes).
4. **Invoke** tools through MCP when the model selects them.
5. **Handle errors** — Timeouts, server restarts, and partial failures without crashing the agent loop.

## Relation to native tools

From the model’s perspective, MCP tools look like any other tools. Prefer MCP when:

- The capability is maintained upstream as an MCP server.
- You want process isolation or language interop.

Prefer **native tools** when you need minimal latency, tight in-process typing, or deep integration with app internals.

## Configuration (conceptual)

Typical settings per server:

- Command or URL, headers, env vars, auth tokens.
- Optional: allowed tool name patterns, rate limits.

Always review **permissions** before connecting a server in production.
