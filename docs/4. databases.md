# Databases

**Databases** (and storage backends) persist what must survive across HTTP requests and process restarts: sessions, messages, user memory, eval traces, and optional knowledge indices.

CentrAI Agent uses a **storage abstraction** so implementations can target SQLite, PostgreSQL, Redis, cloud document stores, or in-memory stores for tests.

## What gets stored

| Kind | Typical use |
|------|-------------|
| **Sessions** | Metadata: ids, user id, created/updated, status |
| **Messages / runs** | Conversation history, tool calls, outputs (for replay and audit) |
| **Session state** | Arbitrary JSON state the agent or tools update across turns |
| **User memory** | Long-lived facts or preferences (optional; privacy-sensitive) |
| **Knowledge** | Embeddings and chunks for RAG (optional; may live in a vector DB) |

Not every deployment needs every feature—embed a minimal adapter for sessions-only if that is enough.

## Adapter contract (conceptual)

A storage layer should support operations such as:

- Upsert session; append messages; load history by session id.
- Merge or replace session state (see [5. session-management](5.%20session-management.md)).
- Optional: prune old runs, export for compliance.

## Choosing a backend

- **SQLite / file DB** — Simple local or single-node deployments.
- **PostgreSQL** — Durable, queryable history; good for multi-tenant apps.
- **Redis** — Fast ephemeral or short-TTL session cache layered on top of durable storage.
- **In-memory** — Unit tests and demos only.

## Security

- Store **PII** only under policy; encrypt at rest where required.
- Separate **tenant** or `user_id` in keys/queries to avoid cross-user leakage.
