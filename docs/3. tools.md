# Tools

**Tools** are capabilities the model can invoke: HTTP APIs, databases, filesystem (where allowed), code execution, search, and more. The agent runtime exposes tools to the model as **schemas** (name, description, parameters) and **executes** them when the model requests a call.

## Tool definition

A minimal tool has:

- **Name** — Stable identifier; must match what the model emits.
- **Description** — Helps the model choose when to use it.
- **Parameters** — Usually JSON Schema (or provider-native equivalent) for arguments.
- **Handler** — Function that runs with validated args and returns a string or structured result.

Tools may be synchronous or async; the runtime should support timeouts and cancellation.

## Toolkits

A **toolkit** groups related tools (e.g. “GitHub”, “Slack”, “internal CRM”) so agents can enable or disable whole bundles. This keeps prompts smaller and reduces accidental tool use.

## Execution pipeline

1. Model returns `tool_calls`.
2. Runtime validates and may apply **middleware** (logging, redaction, rate limits).
3. Handler runs in a controlled context (see [session](5.%20session-management.md) for injecting user/session ids).
4. Result is serialized and sent back as a tool message.

## Safety and permissions

Recommended patterns:

- **Allowlists** — Only register tools that are appropriate for the deployment.
- **Dependency injection** — Pass secrets and clients via runtime context, not global state.
- **Human-in-the-loop** — Optional approval steps before destructive or sensitive tools run.

## MCP

[Model Context Protocol](6.%20mcps.md) servers can **expose** tools to the runtime; those tools are registered like native tools after discovery.
