# Models

The **model** is the reasoning engine behind an agent. In CentrAI Agent, the runtime talks to models through a **provider abstraction** so you can use OpenAI-compatible APIs, Anthropic, local servers (Ollama, vLLM), or unified gateways (LiteLLM, etc.) without rewriting agent logic.

## Responsibilities

| Layer | Responsibility |
|-------|------------------|
| **Model adapter** | Serialize messages, tool schemas, and options; parse responses and tool calls |
| **Agent runtime** | Loop, history, tool execution, session—**not** raw HTTP details |

## Message shape

Agents typically exchange a **chat transcript**:

- **System** — Instructions, policy, optional skill text.
- **User** — End-user or upstream messages.
- **Assistant** — Model outputs, including optional `tool_calls`.
- **Tool** — Results from executed tools (often correlated by `tool_call_id`).

Multimodal inputs (images, audio, files) are represented as structured parts on messages where the provider supports them.

## Tool calling

Modern chat models expose **function calling** (or equivalent): the model returns structured tool invocations (name + arguments). The runtime:

1. Validates arguments against the tool schema.
2. Runs the tool.
3. Appends a tool result message and continues the loop.

Fallback behavior (retries, alternate models) is optional configuration on the agent or model stack.

## Configuration knobs (typical)

- **Model id** — e.g. `gpt-4.1`, `claude-sonnet-4`, local model name.
- **Temperature / max tokens** — Generation controls.
- **Streaming** — Token or chunk streams for UX and lower latency to first byte.
- **Timeouts & retries** — Resilience for flaky networks or rate limits.
