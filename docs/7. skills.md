# Skills

**Skills** are reusable bundles of **instructions**, optional **scripts**, and **metadata** that teach an agent how to perform a task well—similar in spirit to “skills” in editor/agent products and to documented patterns in frameworks like Agno.

They are **not** a replacement for tools: skills tell the model *how* to use tools and in what order; tools *do* the work.

## What a skill contains

| Piece | Role |
|-------|------|
| **Instructions** | Markdown or text merged into system or developer context |
| **Constraints** | Style, output format, safety rules for this domain |
| **Optional assets** | Examples, checklists, small reference data |
| **Optional scripts** | Helper commands the runtime may execute in a sandbox (if enabled) |

## Activation

Skills can be:

- **Always on** for a given agent profile.
- **Selected** by name when the user or router picks a task type.
- **Discovered** dynamically (e.g. from a directory or registry) and summarized in the prompt so the model can request one.

Keep skills **focused**: one skill per workflow (e.g. “code review”, “incident triage”) works better than one giant skill.

## Versioning

Treat skills like code: version them, review changes, and test against representative prompts. Breaking changes to instructions can alter behavior silently.

## Relation to other concepts

- [Tools](3.%20tools.md) provide capabilities; skills describe *procedure* and *policy*.
- [MCP](6.%20mcps.md) may supply tools; skills explain when and how to combine them.
- Long-term preferences may still live in [session](5.%20session-management.md) or user memory—not in skills.
