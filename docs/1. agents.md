# Agents

CentrAI Agent is an **open-source agent runtime**: a small framework for building programs where a language model plans actions, calls tools, and repeats until a task completes or a stop condition is met.

The mental model matches the idea that an agent is a **stateful control loop around a stateless model**—the model proposes steps; the runtime persists session state, routes tool calls, and enforces guardrails.

## What is an agent?

An **agent** is a configured runtime object that:

1. **Receives** user input (and optional structured context).
2. **Builds** a prompt from instructions, memory, session state, and tool definitions.
3. **Invokes** a [model](2.%20models.md) (LLM or compatible API).
4. **Interprets** the response: final answer, or one or more [tool](3.%20tools.md) calls.
5. **Executes** tools, feeds results back into the loop, and continues until done.

You can attach [databases](4.%20databases.md) for persistence, [session](5.%20session-management.md) tracking, [MCP](6.%20mcps.md) servers as tool providers, and [skills](7.%20skills.md) as reusable instruction or capability bundles.

## Core runtime loop

At minimum, each **turn** of the loop does:

```
input + context → model → (assistant message | tool calls)
tool calls → tool results → model → … → final output
```

The runtime is responsible for:

| Concern | Role |
|--------|------|
| **Orchestration** | Max iterations, cancellation, streaming, errors |
| **State** | Session id, conversation history, optional agent/session state |
| **Tools** | Registration, schemas, execution, timeouts, permissions |
| **Safety** | Guardrails, approvals, content filters (optional) |

## Beyond a single agent

Larger systems usually compose:

- **Teams** — Multiple agents with roles, handoffs, or a coordinator pattern.
- **Workflows** — Deterministic or semi-deterministic graphs (steps, branches, parallel blocks) that call agents or functions in order.

CentrAI Agent treats single-agent runs as the foundation; teams and workflows are optional layers on top.

## Design goals

- **Portable**: clear interfaces for models, storage, and tools so you can swap providers.
- **Observable**: structured events for logging, tracing, and debugging.
- **Extensible**: hooks for pre/post run, tool middleware, and custom stop conditions.
