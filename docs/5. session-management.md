# Session management

A **session** is the unit of continuity for an agent: it ties together identity, conversation history, and optional **session state** across multiple turns.

## Identifiers

| Field | Purpose |
|-------|---------|
| **`session_id`** | Correlates all messages and state for one chat or task |
| **`user_id`** | Stable end-user or tenant subject (for memory and ACLs) |
| **`agent_id` / `run_id`** | Optional; useful for multi-agent systems and tracing |

Generate `session_id` on first contact or accept it from the client for reconnecting clients.

## Conversation history

The runtime loads recent messages from [storage](4.%20databases.md), applies optional limits (token budget, last N turns), and may inject summaries for long threads.

## Session state

**Session state** is structured data (e.g. JSON) that:

- Survives across turns in the same session.
- Can be merged with incoming state on each run.
- May be exposed to the model (as context) and/or updated by tools (“agentic state”).

Use it for wizard flows, collected form fields, or pointers to external resources—**not** as a substitute for durable secrets (use a vault).

## Caching

For low latency, optionally cache the active session in memory with invalidation on write. Always persist the source of truth to the database for reliability.

## Lifecycle

Typical flow:

1. Client sends message + `session_id` (or creates one).
2. Runtime loads history and state.
3. Agent runs the model loop; appends new messages; persists updates.
4. Client receives streamed or final output.
