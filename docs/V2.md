# Version 2 — standards, integration, and durable operation

This document scopes the **second major milestone** after [MVP](MVP.md): a more **standard** surface for hosts, **deeper integration** options (protocols and extension points), and **production-grade persistence**—without requiring the full “platform” vision (multi-agent workflows, built-in multi-tenant SaaS) reserved for later phases. It aligns with [8. architecture](8.%20architecture.md) and [9. code-structure](9.%20code-structure.md); foundation topics remain in [1. agents](1.%20agents.md)–[7. skills](7.%20skills.md).

**Status:** **Roadmap / target scope**—implement incrementally after MVP acceptance criteria are met. Individual items may ship as minor releases; this file defines the **thematic bar** for a “v2” release tag when **a majority of** the [integration checklist](#integration-checklist-release-readiness-for-v2) items are satisfied (use judgment for interdependent work, e.g. MCP before relying on remote tools in demos).

**Relationship to v1:** [MVP](MVP.md) stays the baseline for a first shippable runtime. Version 2 **extends** that baseline; it does not redefine the core run loop contract (`Runner`, streaming model, tool loop, `session.Store`) unless called out explicitly with doc updates.

**Streaming rule (agent run):** Every **Run** drives the model through a **streaming** API path only—token/chunk (or equivalent delta) events through to a completed assistant step for the tool loop. There is **no** v2 requirement for a **non-streaming model completion** HTTP/gRPC route or `model.Client` mode for normal agent execution. HTTP **error responses** (4xx/5xx JSON or gRPC status) remain synchronous as usual; only **successful model output** is stream-shaped. This matches [MVP streaming semantics](MVP.md#streaming-semantics-mvp) and tightens the **wire** surface accordingly.

## Goals

| Goal | What “done” means for v2 |
|------|---------------------------|
| **Standard external surfaces** | **Documented** integration paths: stable **Go public API** (packages intended for importers; versioning policy in README or `agent` package docs). Where an **optional process boundary** exists (HTTP or gRPC server), **OpenAPI / protobuf** or equivalent machine-readable contract is published or generated, with explicit **error shapes** and **streaming-only** semantics for **successful** model output. See [HTTP, gRPC, and AI UI stacks (v2)](#http-grpc-and-ai-ui-stacks-v2) for **wire API families** (chat-completions-style streaming, **OpenAI Responses API** streaming, etc.). |
| **MCP first-class** | **`internal/mcp`** (or equivalent) implements an MCP **client** (the architecture doc may call this boundary `mcp.Host`—same responsibility): discover tools from configured servers, map remote tools into the same **`tool.Registry`** execution path as native tools ([6. mcps](6.%20mcps.md)). Configuration is explicit (URLs, auth hooks, timeouts)—no ambient discovery without admin intent. |
| **Skills loader** | **`skill.Loader`** resolves named or path-based skills into instruction fragments consumed by **`prompt`** ([7. skills](7.%20skills.md)): at least **filesystem** resolution with deterministic ordering; optional registry URI later. |
| **Durable persistence by default for server deployments** | **`session.Store`** backed by **SQLite** (or equivalent single-file SQL) is **supported and tested** for typical single-node deploys; **PostgreSQL** adapter is available or clearly documented as the path for multi-tenant / HA apps ([4. databases](4.%20databases.md)). **Migrations** or versioned schema steps are defined for any SQL backend shipped in-tree. **In-memory** remains valid for tests and demos. |
| **Multi-provider models** | At least **two** concrete **streaming** model implementations in-tree—e.g. **OpenAI [Responses API](https://developers.openai.com/api/reference/responses/overview)** (streaming) **and** a second profile (OpenAI-compatible **chat completions** streaming, or another vendor)—sharing **`model.Client`** semantics so hosts can switch providers without changing `agent` ([2. models](2.%20models.md)). |
| **Ingress option** | An **optional** HTTP or gRPC **server** package or `cmd` entrypoint that maps requests to **`Runner`**—auth, rate limits, and tenancy remain **host-owned**, but the **boundary** is standard and documented ([8. architecture](8.%20architecture.md) deployment shapes). Routes are grouped and described in **OpenAPI** (REST) or **protobuf** + **gRPC** service defs; **successful** runs **stream** assistant output (and tool events as defined by the route profile); **errors** use ordinary non-stream HTTP/gRPC error bodies. |
| **Observability** | **Correlation ids** (trace id) propagated from ingress through model and tool spans; **structured lifecycle events** stable enough for log aggregation; optional **OpenTelemetry** hooks or a narrow interface for tracing without forcing a single vendor. |
| **Operational safety hooks** | **Tool middleware** or equivalent for logging, **redaction**, and **policy** (e.g. blocklisted tool names or argument patterns); not a full governance product, but enough for regulated integrations ([3. tools](3.%20tools.md)). |

## Out of scope for v2 (defer)

| Area | Rationale |
|------|-----------|
| **Teams / multi-agent workflows** | Composes multiple `Runner` instances; keep as a **separate package** later ([8. architecture § Future extensions](8.%20architecture.md#future-extensions-non-foundation)). |
| **Built-in vector RAG / embeddings** | Optional knowledge indices belong in product layers or adapters; keep **storage** focused on sessions, messages, and state ([4. databases](4.%20databases.md)). |
| **Full SaaS control plane** | Billing, org management, and global policy engines are **host** concerns; v2 only offers **hooks** and clear boundaries. |
| **Human-in-the-loop product UI** | Approval queues and review dashboards are out of scope; **pause/resume** or callback hooks may appear as optional extension points if needed. |
| **Shipping a React/Next.js app or TS component library** | CentrAI Agent stays **Go**-first; **compatibility** with popular AI frontends is **contract and documentation** only (see below). Hosts may embed [AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/overview) or [AI Elements](https://github.com/vercel/ai-elements) in their own repos. |
| **Non-streaming model completion API** | No first-class HTTP/gRPC route or **`model.Client`** mode that returns a **full model result without a stream** for agent **Run** execution—same as [MVP](MVP.md) (batch/non-streaming completion not required). Callers that need a single blob can **buffer** the stream client-side or in a host adapter. |

## Persistence expectations (v2)

- **Durability**: Restarts must not lose conversation history for deployments that opt into **SQL** backends; document **backup** and **restore** expectations per backend (e.g. SQLite file copy vs Postgres dump).
- **Session state**: **`session.Store`** continues to support merge/replace of **session state** JSON as in [5. session-management](5.%20session-management.md); v2 adapters must not drop fields silently on merge.
- **Redis**: Optional **cache** or TTL layer **on top of** durable store is allowed; not a substitute for append-only message history unless explicitly documented as ephemeral mode.

## Documentation expectations (v2)

Documentation is part of the **“standards”** theme: integrators and operators should not need to read the whole codebase to adopt v2 features safely. The following applies in addition to the general doc hygiene in [9. code-structure](9.%20code-structure.md) (thin `cmd`, config layers, package layout).

### Audiences

| Audience | What they need |
|----------|----------------|
| **Go library users** | Which **import paths** are **stable**, how **versioning** and breaking changes are signaled, and short **integration examples** (construct `Runner`, `Store`, `model.Client`). |
| **HTTP/gRPC consumers** | **Machine-readable contract** (OpenAPI or protobuf) plus human **overview**: auth assumptions, **error model**, and **streaming** behavior. For **AI UI stacks**, see [HTTP, gRPC, and AI UI stacks (v2)](#http-grpc-and-ai-ui-stacks-v2). |
| **Operators** | **Deployment** notes per optional ingress binary: ports, health/readiness if present, **graceful shutdown**, and where **structured logs** go. |
| **Contributors** | **Design docs** (`docs/`) stay aligned with code; **package map** and dependency rules in [9. code-structure](9.%20code-structure.md) remain accurate when packages are added. |

### HTTP, gRPC, and AI UI stacks (v2)

This project does **not** ship TypeScript UI. v2 instead requires **clear wire contracts** so hosts can connect browsers and other agents to the same **Run** semantics as the Go library.

**Machine-readable API description**

- **REST**: **OpenAPI 3.x** (or newer) for every HTTP route the optional ingress exposes—paths, request/response schemas, and **documented** error bodies.
- **gRPC**: **`.proto` files** + generated reference or buf-compatible docs for services that map to **`Runner`** (messages in/out, metadata for trace/session ids).

**Route families (document which ones exist in-tree)**

| Family | Purpose | Notes for v2 docs |
|--------|---------|---------------------|
| **Chat completion–style streaming** | **Streaming** path for **token/chunk** assistant output and, when applicable, **tool-call** steps exposed on the wire | Align field names and stream framing with a **named profile** (e.g. OpenAI **Chat Completions** SSE/chunked bodies) so gateways and proxies behave predictably. |
| **OpenAI Responses API (streaming)** | **Required** in-tree support for at least one implementation that uses OpenAI’s **[Responses API](https://developers.openai.com/api/reference/responses/overview)** with **streaming** (stateful responses, tools, streaming events per OpenAI docs). Map deltas/events to the same **`Runner`** assistant-step + tool-loop contract as chat-completions streaming. **Non-streaming** Responses requests are **not** a supported CentrAI path for **Run**. |
| **Non-streaming model completion** | *Not supported* for agent runs | Do **not** document a “completion without stream” model endpoint as part of CentrAI’s agent surface; see the **Streaming rule (agent run)** blurb at the top of this document. |

**OpenAPI** should tag which operations are **SSE/streaming** vs **error-only** responses so clients never assume a synchronous model body for success paths.

**Agent → agent (machine consumers)**

- **Not** the same as multi-agent *orchestration* in-repo ([Out of scope](#out-of-scope-for-v2-defer) for “teams/workflows”).
- **Is** supported as a **documented HTTP or gRPC client** story: one deployment calls another’s **Run** endpoint with **session/message** payloads and **correlation** (trace id, optional `session_id`). Callers may be another LLM stack, a workflow engine, or MCP-adjacent glue—**auth and policy stay caller/ingress responsibility**.

**Compatibility with [AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/overview) and [AI Elements](https://github.com/vercel/ai-elements)**

- **[AI SDK UI](https://ai-sdk.dev/docs/ai-sdk-ui/overview)** exposes hooks such as **`useChat`**, **`useCompletion`**, and **`useObject`** for streaming UIs; it expects backends to follow Vercel AI SDK **transport and stream protocol** rules (see [Transport](https://ai-sdk.dev/docs/ai-sdk-ui/transport), [Stream protocol](https://ai-sdk.dev/docs/ai-sdk-ui/stream-protocol), and [Reading UI message streams](https://ai-sdk.dev/docs/ai-sdk-ui/reading-ui-message-streams)). That matches CentrAI’s **streaming-first** ingress; it does **not** require CentrAI to expose a non-streaming LLM route.
- **[AI Elements](https://github.com/vercel/ai-elements)** is a **React** component library (shadcn/ui–based) for AI-native layouts; it composes with **`@ai-sdk/react`** patterns and the same **streaming** assumptions as above—it does not define a separate CentrAI-specific protocol.

**v2 documentation obligation**

- Publish a **compatibility matrix**: which **ingress** routes and **stream encodings** match AI SDK **UIMessage** / **Data Stream** expectations **natively**, versus **adapter** patterns (e.g. a thin Next.js **Route Handler** that translates CentrAI’s OpenAPI stream into the AI SDK transport).
- State **tested** combinations when possible (e.g. “`useChat` + default transport against `/v1/...`”) so integrators avoid guesswork.

### Design docs (`docs/`)

- **Update foundation docs when behavior lands**—do not leave MCP, skills, or SQL stores described only in issues or PR text. Prefer extending [6. mcps](6.%20mcps.md), [7. skills](7.%20skills.md), [4. databases](4.%20databases.md), [5. session-management](5.%20session-management.md), and [8. architecture](8.%20architecture.md) over adding many one-off files.
- **Single source of truth** for layers and interfaces remains [8. architecture](8.%20architecture.md) and [9. code-structure](9.%20code-structure.md); if the import graph or public surface changes, update those in the **same change set** as the code when practical.
- **Links**: use **relative** paths and `%20` for spaces in filenames, consistent with existing docs ([README](../README.md) index when adding a **new** top-level `docs/*.md` file).
- **Terminology**: reuse the same terms across docs—**Run**, **tool loop**, **`session.Store`**, **streaming** assistant step—so `docs/8` and `docs/9` do not contradict each other.

### Go API and package comments

- **`godoc`-visible** package comments for every **public** package intended for external import; the **first sentence** should state purpose and stability (e.g. “stable”, “experimental”) where the project distinguishes them.
- If the repo introduces **`pkg/`** for stable DTOs or API types ([9. code-structure §1](9.%20code-structure.md#1-full-repository-layout-recommended)), document that layout in **README** and in [9. code-structure](9.%20code-structure.md) so importers know what is guaranteed.

### Configuration and operations

- **Tables** of **environment variables** and **flags** for each `cmd/*` entrypoint that ships as part of v2 (CLI and optional server)—defaults, required vs optional, and **secrets** called out as “never log”.
- **Persistence**: for each **SQL** backend shipped in-tree, document **schema version**, how **migrations** are applied (automatic vs operator step), and **backup/restore** in one place (can be a subsection of [4. databases](4.%20databases.md) or a short `docs/` operations note linked from there).
- **MCP**: document **transport(s)** supported, **tool naming** when merging with native tools (prefixes, collisions), and **timeouts**—either in [6. mcps](6.%20mcps.md) or linked from it.

### Observability

- A **stable list of lifecycle event names** (or enum values) and **recommended fields** (session id, run id, trace id, latency, error class)—in [8. architecture](8.%20architecture.md) observability section or a small `docs/` subsection—so log pipelines can key off documented strings.
- If **W3C Trace Context** or similar headers are supported at ingress, document **header names** and propagation rules.

### Releases and compatibility

- **Versioning policy** stated in **README** or module documentation: Go **module path** rules for breaking changes (e.g. `/v2` suffix) and what counts as a **compatible** change for library users.
- **CHANGELOG** or **release notes** for tagged releases that touch **public** APIs or **wire** contracts, with **migration** hints when behavior changes.

## Integration checklist (release readiness for v2)

Use this list to decide when the **v2** theme is met for a release tag (in addition to MVP checks still relevant).

- [ ] MCP client lands; remote tools route through **`tool.Registry`** with tests that do not require live network (fakes/fixtures).
- [ ] **Skills** load from filesystem (or documented equivalent) into **`prompt`** without duplicating skill text in app code.
- [ ] **SQLite** (and/or **PostgreSQL**) **`session.Store`** implementation(s) with **migrations** or versioned schema and tests.
- [ ] **Multi-provider models** ([Goals](#goals)): at least **two** **streaming** **`model.Client`** implementations with shared contract tests—**including** OpenAI **[Responses API](https://developers.openai.com/api/reference/responses/overview)** (streaming) mapped to `Runner` steps, plus a second profile (e.g. chat-completions streaming); tests without live network where possible.
- [ ] Optional **ingress** binary or package documented with request/response contract (OpenAPI or proto), **route families** ([chat completion–style streaming](#http-grpc-and-ai-ui-stacks-v2) and Responses streaming), and **streaming-only** success paths for model output (no non-streaming completion route for **Run**).
- [ ] **AI SDK UI / AI Elements**: [compatibility matrix](#http-grpc-and-ai-ui-stacks-v2) or **adapter** recipe (e.g. Next.js route translating CentrAI streams to [AI SDK transport](https://ai-sdk.dev/docs/ai-sdk-ui/transport)) documented for at least one recommended integration path.
- [ ] **Observability**: trace/correlation id threading documented; structured events listed in architecture or package docs.
- [ ] **Public Go API**: import paths intended for external use are identified; **versioning policy** (e.g. semver / v2 module path rules) documented in README or package docs, matching **Standard external surfaces** above.
- [ ] **Tool middleware** (or equivalent) for logging, redaction, and basic policy hooks—documented and covered by tests.
- [ ] **README** and this file updated if scope shifts; no contradiction with [8. architecture](8.%20architecture.md) / [9. code-structure](9.%20code-structure.md).
- [ ] **Documentation expectations (v2)** satisfied for shipped features: foundation docs updated where behavior changed; env/flags tables for supported `cmd` paths; **CHANGELOG** or release notes for API/wire changes; **lifecycle event** names/fields documented for observability.
